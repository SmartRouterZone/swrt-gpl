# S20P AX6000 SWRT 适配记录

## 当前状态

本仓库 `/home/jack/swrt-gpl` 已有 MT7986/MT7981 平台支持，S20P AX6000 适配先按最小 bring-up 路径接入构建系统。

本次新增内容：

- 新增 `SWRT-S20P-AX6000` 构建 profile，预期构建命令：`make swrt-s20p-ax6000`。
- 新增 kernel model：`CONFIG_MODEL_S20PAX6000`。
- 新增 DTS：`mt7986a-S20P-AX6000.dts`。
- 复用 MT7986 vendor WiFi/warp/conninfra 编译入口。
- 用户态新增 `S20P-AX6000` / `S20PAX6000` model 映射。
- 先按 MT7986 + MT7531 + 7 个有线口的板型接入 `mt7986_mt7531.c`。

## 硬件摘要

- SoC: MediaTek MT7986A, Cortex-A53, 4 cores
- RAM: DDR4 2 GiB
- Storage: eMMC 128 GiB, HS400
- Switch: MediaTek MT7531AE
- Ethernet: 5 x 1G LAN, 2 x 2.5G RTL8221B-VB-CG
- WiFi: MT7975N 2.4 GHz 4x4, MT7975PN 5 GHz 4x4
- USB: 1 x USB 3.0
- M.2: socket present, PCIe currently enabled in DTS

## DTS 关键点

网络拓扑按 S20P 实机/OpenWrt 适配记录整理：

- `gmac0` 固定 2500base-x 连接 MT7531 CPU port。
- MT7531 port 0-4 映射 `lan5 lan4 lan3 lan2 lan1`。
- MT7531 port 5 映射 `lan6`，外接 RTL8221B-VB-CG，MDIO address 5。
- `gmac1` 映射独立 2.5G `eth1`，外接 RTL8221B-VB-CG，MDIO address 7。
- HNAT 暂按 `wan = eth1`、`lan = lan`、`ppd = eth0`、2 GMAC / 2 PPE 配置。

## 重要注意事项

### 1. RTL8221B-VB-CG 仍是最高风险点

当前 5.4 tree 的 `drivers/net/phy/realtek.c` 没有完整 RTL8221B 支持，DTS 只能声明 C45 PHY，不能保证 2.5G 链路正常。后续必须移植或 backport RTL8221B 相关 PHY 支持，并验证：

- `eth1` 2.5G 协商、link up/down、中断；
- `lan6` 2.5G 协商、link up/down、中断；
- 1G/2.5G/强制速率切换；
- HNAT/WED offload 下的稳定性。

### 2. WiFi EEPROM/校准数据还没有完成真实 eMMC 读取适配

OpenWrt 适配里可以用 nvmem/block partition 表达 factory 数据，但这个 SWRT 5.4 vendor tree 未确认支持同等 block nvmem 布局。当前 DTS 先启用 `wbsys` 和 WiFi pinctrl，未声明 `mediatek,mtd-eeprom`。

后续要二选一验证：

- 移植 eMMC factory 到 WiFi EEPROM 的真实读取路径；
- 或继续沿用启动脚本从 eMMC factory 提取并覆盖 vendor 默认 EEPROM bin 的方式。

### 3. eMMC 升级路径未完成

已在 profile 中启用 `EMMC=y`，但 SWRT 现有 `RTCONFIG_EMMC` 升级路径主要服务其它机型。S20P 还需要确认：

- 目标镜像格式；
- kernel/rootfs 写入分区名；
- `block-write` 偏移；
- recovery/failsafe 行为；
- 保留 factory、bootloader、env 分区。

### 4. 用户态端口映射只是第一版

`mt7986_mt7531.c` 已加入 S20P 的 7 口模型和 PHY 映射：

- `lan6 -> external PHY 5`
- `lan5 -> MT7531 port 0`
- `lan4 -> MT7531 port 1`
- `lan3 -> MT7531 port 2`
- `lan2 -> MT7531 port 3`
- `lan1 -> MT7531 port 4`
- `wan/eth1 -> external PHY 7`

但 SWRT UI、双 WAN、IPTV、LED、force link、EasyMesh 等逻辑分散在多个用户态文件，首次编译/运行后需要继续根据错误和实机行为收敛。

## 建议排查顺序

1. 先做静态检查：`git diff --check`。
2. 再只编译 device tree / kernel config 生成阶段，确认 `CONFIG_MODEL_S20PAX6000` 能选中 `mt7986a-S20P-AX6000.dtb`。
3. 编译失败时优先看 model 宏遗漏，其次看 DTS label/属性是否不被 5.4 vendor tree 支持。
4. 首次上机只验证串口、eMMC rootfs、eth0/MT7531、eth1/lan6 PHY 初始化。
5. 再处理 WiFi EEPROM 和 vendor mt_wifi 启动。
6. 最后处理 HNAT/WED、升级写入、LED/按键、UI 端口显示。

## RTL8221B 与 EEPROM 补充记录

本次继续补了最后两项的最小实现：

- `drivers/net/phy/realtek.c` 增加 `RTL_8221B_VB_CG = 0x001cc849`，支持 C22/C45 识别。
- RTL8221B-VB-CG 复用当前 5.4 tree 已有的 RTL8125 2.5G 能力、自协商、状态读取和 MMD 读写逻辑。
- 增加 RTL8221B C45 VEND2 link interrupt 的 ack/config 逻辑，状态机触发沿用 PHY core 默认处理。
- `init_wl()` 中对 `S20PAX6000` 增加 eMMC factory 提取：优先读取 `/dev/disk/by-partlabel/factory`，其次 `/dev/disk/by-partlabel/Factory`，输出 `/lib/firmware/e2p`，大小沿用 MT7986 DBDC 的 `720896` 字节。

这仍然不是最终实机闭环。RTL8221B 需要确认内核枚举时实际 `phy_id`/`c45_ids.device_ids[1]` 是否为 `0x001cc849`；EEPROM 需要确认 SWRT 运行时是否存在 `/dev/disk/by-partlabel/factory`，以及 mt_wifi 是否接受 `/lib/firmware/e2p`。
